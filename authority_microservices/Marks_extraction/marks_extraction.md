# Marks Extraction Service - Architecture & Implementation Plan

This document outlines the architectural plan for building the Marks Extraction system as a modular Python service inside your AI backend. It focuses on file organization, the service design, local manual testing, and the future API integration.

---

## 1. Folder Structure & File Naming
To keep your codebase clean and modular, we will create a dedicated module folder specifically for this task inside your `authority_ai_service_backend`.

```text
authority_ai_service_backend/
│
└── Marks_extraction/                  # The main module directory
    │
    ├── __init__.py                    # Makes this folder an importable Python module
    │
    ├── extractor_service.py           # Core logic: The main function you will export
    ├── pdf_processor.py               # Helper: Handles the complex Camelot multipage logic
    ├── csv_excel_processor.py         # Helper: Handles Pandas logic for spreadsheets
    │
    └── test_extractor.py              # The local manual testing script
```

---

## 2. File Responsibilities (The Blueprint)

Here is the plan for what each file is responsible for.

### `extractor_service.py`
This is the "Director". Your future API route will only ever import from this file.
*   **Exported Function:** `extract_marks_from_document(file_path, file_type)`
*   **Role:** It looks at the file type, routes it to either the PDF processor or the CSV/Excel processor, normalizes the headers, cleans the grid, and returns the final JSON array.

### `pdf_processor.py`
This handles the most complex, risky logic.
*   **Internal Function:** `process_multipage_pdf(file_path)`
*   **Role:** Uses `camelot-py` to extract grids, forces Page 1 headers onto Page 2+, and stitches everything together using `pd.concat`.

### `csv_excel_processor.py`
This handles the easy formats.
*   **Internal Function:** `process_spreadsheet(file_path)`
*   **Role:** Uses `pandas` to read `.csv` or `.xlsx` files accurately.

## 3. The Local Testing Plan (`test_extractor.py`)

Before hooking this up to the API, we will build a dedicated script for manual local testing.

**How it works:**
1. The `test_extractor.py` file will prompt you to manually submit a document path in your terminal.
2. It uses Python's `input()` function or `sys.argv` to take the absolute path of whatever real document you want to test right now.
3. It imports `extract_marks_from_document` from `extractor_service.py` and passes your manual path to it.
4. It receives the JSON array and prints it beautifully to the console (or saves it to a local `test_output.json`) so you can visually confirm the mapping is 100% accurate.

**Execution:**
You simply open your terminal and run:
`python Marks_extraction/test_extractor.py`
The terminal will ask: `"Enter the absolute path to your document: "`

## 4. Future API Integration Plan

Once local manual testing is successful and you confirm the documents parse perfectly, the integration with your main API hub will be incredibly simple because the service is fully modular.

**The API Flow:**
1. **The Endpoint:** You will create an endpoint (e.g., `POST /api/v1/extract-marks`) in your main FastAPI routing file.
2. **File Reception:** The API will receive the uploaded document via `UploadFile` and temporarily save it to the disk.
3. **The Call:** The API will import and call our service function: 
   `json_data = extract_marks_from_document(temp_file_path, extension)`
4. **The Response:** The API will return the exact JSON payload generated by the service back to the client or the Node.js backend.
   
**Example JSON API Response:**
```json
{
  "status": "success",
  "data": [
    {
      "usn": "1RUA24BCA0001",
      "assignment_10_marks": 8,
      "quiz_10_marks": 7,
      "total_20_marks": 15
    }
  ]
}
```
This ensures that the API layer handles only HTTP/Routing, while the `Marks_extraction` module handles all the heavy lifting.
